# FastAPI 与后端开发学习汇总

本文档用于记录在项目开发过程中积累的核心概念、坑点以及工业界最佳实践。

## 1. 全局异常处理机制
*   **统一返回格式**：无论成功还是失败，给前端返回定式的 JSON（如 `status_code`, `message`, `data`），防止前端解析崩溃。
*   **异常层级拦截**：
    *   `AppException` (自定义业务异常)：负责处理我们主动拦截的逻辑错误（如密码错误、找不到用户等）。
    *   `RequestValidationError`：拦截 Pydantic 参数校验错误，将其格式化成我们需要的 JSON 结构。
    *   `Exception`：作为最后的兜底，拦截所有未知的系统 BUG（如除零、空指针），对外返回统一的 500 错误，并在后台利用 logger 打印真实的堆栈信息。
*   **底层做绝，上层省心**：在类似 `decode_token` 这样的底层工具函数里，如果解析失败或过期，**直接主动抛出自定义异常**（如 `raise TokenInvalidException()`），而不是返回 `None` 交给上层去判断。这样可以避免大量重复代码，防止越权漏洞。

## 2. 数据库异步建表与模型加载
*   **异步建表语法**：在 SQLAlchemy 异步模式下，必须使用 FastAPI 的 `lifespan` 配合 `conn.run_sync(Base.metadata.create_all)` 来建表。
*   **模型的中心化注册**：SQLAlchemy 能否建表，取决于内存中 `Base.metadata` 是否发现了这些类。
    *   **最佳实践**：在 `models/__init__.py` 中把所有表（`Base`, `User` 等）集中 `import`。在 `main.py` 中只要 `from models import Base` 这一句，Python 解析器就会自动加载所有通过 `__init__.py` 暴露的模型，既避免了“导入未使用”的 linter 警告（不需要写 `# noqa`），又保证了表能成功创建。

## 3. 关于异常实例化的语法规范
*   **一定要带括号**：建议使用 `raise TokenInvalidException()` 而不是 `raise TokenInvalidException`。
*   **原因**：虽然两者在无参数时都能运行，但在面向对象的工业规范里，带括号显式表明我们在“实例化”一个异常对象，可读性更强，且未来如果要在异常内部添加定制参数（如 `message="..."`）时不易出错。

## 4. JWT、依赖注入与 tokenUrl
*   **闭环设计**：我们不需要担心恶意构造不带 `user_id` 的 Token。因为合法的 Token 必定是我们自己后端 `login` 接口颁发出去的，只要在生成 Token 时把 `user_id` 强制写入 payload 字典里，解析时就一定能拿到。
*   **依赖集中管理**：不要把获取数据库 `get_db` 和获取用户 `get_current_user` 这类依赖散落在各个角落。建立一个统一的 `utils/dependencies.py` 是很好的结构。
*   **tokenUrl 的本质**：`OAuth2PasswordBearer(tokenUrl="/api/users/login")` 里的这个 URL **不参与**实际的代码权限校验！它仅仅是告诉 FastAPI 自带的 Swagger UI 文档 (`/docs`)，在点击右上角的绿锁 🔒 去获取 Token 方便测试时，该把账号密码提交给哪个接口。

## 5. SQLAlchemy 中 flush, commit 和 refresh 的区别
*   **`flush()` （“用铅笔写答案”）**：将数据推向数据库，生成了 SQL 语句（比如拿到了自增的 ID），**但还没有真正持久化**。此时的数据在事务里是“脏数据”，自己同一个代码上下文里能查到，但外部其他用户的查询绝对看不见。如果报错，数据随风飘散。
*   **`commit()` （“用黑笔涂答题卡并上交”）**：将事务一锤定音，真正写入硬盘持久化！
*   **`refresh()` （“从真实考卷上对答案”）**：在 `commit()` 发生之后，把最新的真实数据（例如数据库底层补充的默认值、自增 ID、`created_at` 时间戳等）强制拉取（`SELECT`），覆盖更新到当前的 Python 对象（如 `new_user` 上），以便接下来返回给前端。

## 6. Pydantic 响应模型与 SQLAlchemy 的 from_attributes
*   **现象**：我们只传了3个字段注册（username, email, password），但最终的 UserResponse 结果却返回了包含 id, created_at, 默认 gender 等 8 个字段。这并非凭空捏造。
*   **本质**：这是由于关系型数据库和 ORM 的机制所致。
    1. 我们传入的 3 个注册字段经过 db.add 和 db.commit 被正式插入到了数据库中。数据库随后利用表定义结构（如 default, autoincrement 规则、当前时间戳等）自己生成了剩下的所有缺省字段。
    2. 通过调用 `db.refresh(new_user)`，new_user 实例在内存中“丰满”了起来，同步了数据库里完整的真实数据。
    3. 响应模型开启了配置项 `model_config = ConfigDict(from_attributes=True)`。这指示 Pydantic 不仅可以从字典中取值，也允许直接读取对象的属性（如 new_user.id、new_user.gender）。因此，Pydantic 顺利从已刷新的数据库实例中“提取”出了所有需要的字段，成功返回！

## 7. 响应模型存在数据库中没有的字段会怎样？
*   **场景**：如果在 UserResponse 中定义了一个数据库 ORM 表里根本不存在的字段（如 nickname）。
*   **结果分两种情况**：
    1. **未提供默认值（严格模式 - 报错拦截）**：像 `nickname: str`。FastAPI 在转换 ORM 对象为 JSON 时，发现对应的底层对象压根没有 `.nickname` 属性。此时直接触发 500 Internal Server Error 并抛出 ResponseValidationError。因为这破坏了你自己承诺的数据结构。
    2. **提供了默认值（宽容模式 - 正常通过）**：像 `nickname: str = "默认"` 或是 `nickname: Optional[str] = None`。在找不到对应属性时，Pydantic 会判定退而求其次，直接采用你在响应模型里设置的“默认值”将其返回给前端。
*   **实际应用**：如果需要响应前端某个复合计算出的属性（例如不需要持久化存储的虚字段），推荐的最佳做法是在 SQLAlchemy 的实体类中加一个 `@property` 函数（例如 `def nickname(self): return "用户_"+str(self.id)`）。这样 Pydantic 就会像读取普通变量一样拿到该属性的值！
## 8. 静态资源托管与“相对路径陷阱”
*   **相对路径陷阱**：在数据库里存储类似 ../images/default.png 的相对路径是非常危险的。前端（如 Vue/React）拿到这串文本后，会相对于前端所在的网页 URL 发起请求（例如 http://localhost:3000/images/default.png），从而导致图片 404 找不到，因为前端根本没有这个服务器物理路径。
*   **正确的工业界做法**：
    1.  **数据库存储网络路由绝对路径**：在 SQLAlchemy 模型默认值中，存入以 / 开头的网络路由路径，例如 default="/static/default.png"。
    2.  **后端挂载静态目录**：在 FastAPI 的 main.py 中，使用 StaticFiles 将本地的物理文件夹（如 images）暴露给网络的 /static 路由。
        代码示例：pp.mount("/static", StaticFiles(directory="images"), name="static")
*   **最终效果**：当前端拿到 /static/default.png 并在浏览器中请求时，请求会发往 http://127.0.0.1:8000/static/default.png，FastAPI 会自动去本地文件系统的 images/ 目录下找到它并返回。
*   **补充**：在大型商业项目中，用户上传的头像等媒体资源通常不会放在本地服务器上，而是会扔给云端的对象存储（如阿里云 OSS、腾讯云 COS），数据库里的 default 则会是一串完整的云端绝对链接（例如 https://oss.xxx.com/default.png）。

## 9. 异常冒泡与全局统一响应：底层抛，顶层包
*   **问题场景**：在 CRUD 函数（如 `authenticate_user`）中遇到密码错误时抛出 `PasswordErrorException`，但函数本身没有 `return`，前端到底能收到什么？
*   **核心机制——异常冒泡**：Python 中的异常会沿着调用栈一直向上抛，直到被某个 `try/except` 捕获，或者到达最顶层被框架捕获。FastAPI 提供了 `@app.exception_handler()` 装饰器，就是在最顶层统一"接盘"。
*   **实际流程**：
    1.  `crud/user.py` 中 `raise PasswordErrorException()` → 函数中断，不返回任何值。
    2.  异常向上冒泡到调用方 `routers/user.py` 的 login 接口函数。
    3.  login 函数也没有捕获，继续向上冒泡到 FastAPI 框架层。
    4.  `main.py` 中注册的 `@app.exception_handler(AppException)` 捕获到该异常，将其格式化为统一的 `ApiResponse` JSON 返回给前端。
*   **前端收到的响应**：
    ```json
    {
      "status_code": 400,
      "message": "用户名或密码错误",
      "data": null
    }
    ```
*   **设计优势**：
    *   **底层做绝，上层省心**：CRUD 层只管判断业务逻辑、抛异常，完全不用关心 HTTP 状态码怎么设、JSON 格式怎么拼。
    *   **统一收口**：所有业务异常的响应格式由 `main.py` 的一个处理器决定，一改全改，不会出现有的接口返回格式不一致的混乱情况。
    *   **安全提示**：登录失败返回"用户名或密码错误"这种模糊信息是工业界标准做法，避免暴露到底是用户名不存在还是密码错了，防止被恶意枚举。
*   **异常冒泡的完整路径（以登录为例）**：
    ```
    authenticate_user() 里 raise PasswordErrorException()
            ↓ 向上抛
    login() 没捕获，自己也"中断"了，后续代码（生成 token、return）不执行
            ↓ 继续向上抛
    FastAPI 框架层 → @app.exception_handler(AppException) 捕获
            ↓ 生成 JSONResponse 返回前端
    ```
*   **如果 router 想自己捕获怎么办？** 可以用 `try/except` 包住调用，但通常不建议——全局处理器已经统一处理了，router 保持干净即可。除非你需要在捕获后做额外逻辑（如记录审计日志、返回不同格式）。

## 10. HTTP 状态码规范：让语义与场景精确匹配
*   **核心原则**：状态码不只是给前端看的数字，它是 HTTP 协议的标准语义，能让前端、网关、日志系统、监控工具都正确理解发生了什么。
*   **本项目规范后的映射**：

    | 异常类 | 场景 | 状态码 | 语义 |
    |--------|------|--------|------|
    | `UserNotFoundException` | 查询不到用户 | 404 | Not Found，资源不存在 |
    | `UserAlreadyExistsException` | 注册用户已存在 | 409 | Conflict，资源冲突 |
    | `EmailAlreadyExistsException` | 注册邮箱已存在 | 409 | Conflict，资源冲突 |
    | `PasswordErrorException` | 登录密码错误 | 401 | Unauthorized，认证失败 |
    | `TokenInvalidException` | Token 无效或过期 | 401 | Unauthorized，认证失败 |
    | `RequestValidationError` | 参数校验失败 | 422 | Unprocessable Entity，参数格式错误 |
    | 未捕获系统异常 | 代码 BUG | 500 | Internal Server Error，服务端错误 |

*   **常见误区纠正**：
    *   ❌ 登录失败用 `400`：`400` 表示"请求语法错误"，而登录密码错误是"身份认证不通过"，应该用 `401`。
    *   ❌ 用户已存在用 `400`：`400` 太泛了。`409 Conflict` 是 HTTP 专门用来表示"资源冲突"的状态码，更精确。
    *   ✅ 记住：`400` 系列是客户端的"错"，但不同"错"有细分。不要乱用 `400` 一刀切。

## 11. Token 管理：双 Token 机制与有状态/无状态方案
*   **为什么需要双 Token？**
    *   `access_token`（访问令牌）：有效期短（15分钟），用于每次 API 请求的身份认证。频繁携带，被盗风险高，所以有效期要短。
    *   `refresh_token`（刷新令牌）：有效期长（7天），只用于在 access_token 过期后换取新的 access_token。不直接用于业务请求，暴露频率低。
    *   **好处**：access_token 被盗了也只用忍15分钟；refresh_token 被盗了可以通过白名单机制吊销。
*   **无状态 JWT 方案（简单但不安全）**：
    *   服务端不存 Token，只靠 JWT 的签名和 `exp` 过期时间来判断有效性。
    *   **致命缺陷**：无法主动吊销。用户修改密码后，攻击者手里的旧 Token 仍然有效直到自然过期。
    *   **适用场景**：内部系统、对安全性要求不高的项目。
*   **有状态 Token 方案（推荐）**：
    *   服务端维护 Token 白名单（数据库或 Redis），记录每个用户当前有效的 refresh_token。
    *   **登录时**：生成双 Token，refresh_token 存入白名单。
    *   **刷新时**：校验 refresh_token 是否在白名单中 → 生成新双 Token → 旧 refresh_token 从白名单移除 → 新 refresh_token 存入白名单。
    *   **改密码/登出时**：把该用户的所有 Token 标记为 `revoked`，立即失效。
    *   **为什么只存 refresh_token？** access_token 只有15分钟寿命，就算被盗影响极小，不需要额外维护成本。
*   **多设备登录的处理**：
    *   支持多设备 → 白名单里保留该用户的多个 refresh_token（每个设备一个）。
    *   单设备登录 → 登录时先清掉该用户所有旧 Token，保证"后登录踢掉先登录"。
*   **Token 存储位置**：
    *   ❌ localStorage：容易受到 XSS 攻击，恶意脚本可以直接读取。
    *   ✅ Cookie（httpOnly + secure）：浏览器自动携带，JS 无法读取，防 XSS。配合 CSRF Token 或 SameSite 属性防 CSRF。
*   **刷新 Token 的两种方案对比**：
    *   **方案A：只换 access_token，refresh_token 固定7天**
        *   优点：安全性高，refresh_token 被盗也只用忍7天。
        *   缺点：用户7天必须重新登录一次，体验差。
    *   **方案B：每次刷新同时换新的 refresh_token（滑动续期）**
        *   优点：用户只要持续使用，就永远不会掉登录，体验好。
        *   缺点：refresh_token 被盗可以一直续期。但配合白名单机制（每次刷新删旧换新），被盗的旧 refresh_token 会立即失效，风险可控。
        *   ✅ **推荐**：在移动端/长期会话场景下，滑动续期是工业界主流做法（如微信、GitHub）。
*   **滑动续期的完整流程**：
    ```
    前端调 /refresh  Header: Bearer <refresh_token>
        ↓
    后端：
        1. 解码 refresh_token（验签名+过期时间）
        2. 查白名单：这个 refresh_token 在不在库里？
        3. 校验通过 → 生成新的双 Token
        4. 从白名单删除旧的 refresh_token
        5. 新的 refresh_token 存入白名单
        6. 返回 {access_token, refresh_token}
    ```
    *   **关键**：每次刷新后，旧的 refresh_token 立即失效（从白名单删除），即使被盗也无法使用。
*   **前端什么时候调用 /refresh？**
    *   **场景1：被动刷新**
        ```
        请求API → 返回401 "Token过期" → 调/refresh → 重试请求
        ```
    *   **场景2：主动刷新（推荐，无感知）**
        ```
        前端在access_token过期前2分钟主动调/refresh
        或：每次页面切换/发请求前检查，快过期就提前刷新
        ```
    *   **场景3：refresh_token 也过期了（用户7天没来）**
        ```
        /refresh 返回401 → 前端清除Token → 跳转登录页
        ```
